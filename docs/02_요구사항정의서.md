## 1. 개요

### 1.1 목적
본 문서는 유플러스아이티가 수행하는 IT 프로젝트(특히 공공/SI 사업)의 전 라이프사이클을 단일 플랫폼에서 관리하기 위한 uplusitPMS의 기능 및 비기능 요구사항을 정의한다. NeoPMS와 WATCH2 제품군의 강점을 통합하여 계획(Waterfall)과 실행(Agile)을 하나의 데이터로 연결하는 통합 프로젝트 운영 표준을 목표로 한다.

### 1.2 범위
프로젝트 기본관리, 일정·공정 진척, 요구사항 관리, 개발공정 도구, 진척(칸반), 품질·산출물, 테스트·결함, 이슈·리스크, 자원, 의사소통·보고, 협업 도구, 방법론·거버넌스, 보안·운영, **AI/지능형 자동화, 통합 알림센터**를 포함한다.

### 1.3 사용자 및 역할
| 역할 | 설명 | 주요 권한 |
|------|------|-----------|
| 시스템관리자 | 시스템 전역 설정·코드·백업/복원 | 전체 |
| PMO | 멀티 프로젝트 모니터링, 표준 방법론·산출물 정의 | 거버넌스·조회 |
| PM | 단일 프로젝트 일정·자원·진척·산출물·결재 총괄 | 프로젝트 전체 |
| 팀장(PL) | 팀 단위 WBS·산출물·보고서·1차 검토 | 팀 단위 |
| 팀원(개발자/QA/DBA) | 본인 태스크 진척·산출물·결함 조치 | 본인 단위 |
| 시험자 | 케이스 수행결과·결함 등록 | 시험 단위 |
| 발주처/감리원 | 진척·산출물·리스크 조회, 결재 최종단계 | 읽기 전용/결재 |

### 1.4 우선순위 표기 (MoSCoW)
**M**(Must, 필수) · **S**(Should, 권장) · **C**(Could, 선택) · **W**(Won't now, 차기)
**출처 표기:** N=NeoPMS, W=WATCH2-Project, D=WATCH2Defect, U=유플러스아이티 신규

## 4. 요구사항 추적성

본 요구사항은 후속 단계에서 다음과 같이 추적한다.

> **요구 ID (REQ-xxx) → 유스케이스 → 설계(화면/API/테이블) → 구현 프로그램 → 테스트 케이스 → 결함**

- 모든 기능 요구는 분석 단계에서 최소 1개 유스케이스로 매핑한다.
- 요구사항 관리 모듈(FR-04)이 베이스라인·변경관리의 단일 기준점이 된다.
- 테스트 단계의 RTM(REQ-TST-07)이 요구↔테스트 커버리지를 자동 검증한다.

## 5. 가정 및 제약

- 주 적용 대상은 공공/SI 사업이며, 계획(Waterfall)과 실행(Agile)을 병행한다.
- 폐쇄망(망분리) 운영을 전제로 하며 외부 SaaS 의존을 최소화한다.
- 기술 스택은 설계 단계에서 확정하되, 벤치마크 기준 PostgreSQL·웹표준·Docker를 우선 검토한다.

## 6. 미결 사항 (Open Issues)

- 소스코드 제너레이터(REQ-DEV-04) 채택 범위 — 내재화 vs 외부도구 연계.
- 메신저(REQ-CLB-03) 구현 방식 — 자체 개발 vs Rocket.Chat류 오픈소스 연동(개선분석 권고: 온프레미스 RC 연동).
- AI/자동화(FR-15) 적용 범위 — 온프레미스 sLLM 모델 등급(경량/표준)과 GPU 사양은 발주처 인프라 제약에 따라 설계 단계 확정.
- 프론트엔드 — 상용 RIA(WebSquare) vs 표준 SPA(React/Vue) 라이선스·공수 트레이드오프(개선분석 §5 참조).
