## 1. 개요

### 1.1 목적
요구사항정의서(SRS v0.2)의 기능 요구를 **유스케이스·업무 프로세스·개념 데이터 모델·화면 흐름**으로 변환하여 설계 단계의 입력을 확정한다.

### 1.2 범위 및 심화 기준
SRS의 16개 기능영역(FR-01~16) 전체를 유스케이스로 매핑하되, **MVP(우선순위 Must) 모듈을 상세 명세 수준으로 심화**한다. Should/Could 항목은 유스케이스 목록·추적표 수준으로 다루고 설계/구현 단계에서 구체화한다.

**MVP 심화 대상(Must 중심):** 대시보드, 프로젝트 기본관리, 일정/WBS·진척, 요구사항 관리, 진척(칸반)·계획↔실행 브리지, 품질/산출물, 테스트/결함, 이슈, 자원, 의사소통/보고, 협업, RBAC/감사, 통합 알림.

### 1.3 분석 방법
액터 식별 → 유스케이스 도출(요구 ID 1:N 매핑) → 핵심 유스케이스 흐름 명세 → 업무 프로세스 흐름 정리 → 개념 데이터 모델 도출 → 화면 흐름 정의 → 추적성 검증. 모든 유스케이스는 최소 1개 이상의 `REQ-xxx`에 연결한다.

### 3.2 핵심 유스케이스 상세 명세 (MVP)

#### UC-DSH-01 통합 대시보드 조회 및 드릴다운
- **주액터:** 전체 / **사전조건:** 로그인, 프로젝트 선택됨
- **기본흐름:** ① 로그인 후 프로젝트 대시보드 진입 → ② 시스템(SVC)이 WBS·이슈·결함·결재 데이터를 자동 집계(SSOT)하여 위젯 표시 → ③ 사용자가 위젯(진척률·품질달성도·이슈/결함처리율·오늘의 작업) 확인 → ④ 위젯 클릭 시 해당 상세 화면으로 드릴다운 이동.
- **대안/예외:** 2a. 데이터 없음 → 빈 상태 안내. 4a. 권한 없는 상세 → 접근 차단.
- **사후조건:** 최신 집계가 화면에 반영. **관련:** REQ-DSH-01,02,03,05.

#### UC-SCH-01 WBS 등록·뷰 전환
- **주액터:** PM/PL / **사전조건:** 프로젝트·방법론(단계) 정의됨
- **기본흐름:** ① WBS 트리에 작업(Task) 계층 등록 → ② 엑셀 템플릿 일괄 등록/다운로드 → ③ 트리/간트/캘린더 뷰 전환 → ④ 단계×업무 가중치 입력(합계 검증) → ⑤ 저장.
- **대안/예외:** 2a. 검증 오류 행 표시. 4a. 가중치 합계 불일치 → 잠금 해제 불가.
- **사후조건:** WBS 기준선 생성, 작업카드 생성 트리거(→UC-BRG-01). **관련:** REQ-SCH-01,07.

#### UC-SCH-02 실적 입력·가중치 진척 자동산정
- **주액터:** PL/TM / **사전조건:** WBS Task에 담당자·가중치 배정
- **기본흐름:** ① 담당자가 Task 실적(실적기간·수치/산출물 기반) 입력 → ② 진척률 상한 규칙 적용 → ③ 단계×업무·depth별 가중치 곱연산으로 상위 진척률 자동 산정 → ④ S-Curve·대시보드·EVM에 반영.
- **대안/예외:** 1a. 다단계 결재형 실적 → 승인 후 반영. 2a. 상한 초과 입력 차단.
- **사후조건:** 프로젝트 진척률 갱신. **관련:** REQ-SCH-02,03,04 / 연계 REQ-EVM-01.

#### UC-RQM-02 요구사항 베이스라인 확정·변경관리
- **주액터:** PM / **사전조건:** 요구사항 등록·"수용" 상태 존재
- **기본흐름:** ① 수용·미확정 요구를 일괄 선택 → ② 베이스라인 확정 → ③ 확정 후 변경 발생 시 변경 화면에서 **사유·근거문서·승인자 필수 입력** → ④ 변경 이력 자동 영구 기록 → ⑤ 확정 요구만 개발/테스트 매핑 대상이 됨.
- **대안/예외:** 3a. 필수 항목 누락 → 저장 차단. 3b. 미승인 변경 → 매핑 무결성 보호로 반영 보류.
- **사후조건:** 베이스라인 버전 증가, RTM 재계산. **관련:** REQ-RQM-02,03 / 연계 REQ-RQM-05.

#### UC-KAN-01 작업카드 진척 등록 (칸반)
- **주액터:** TM / **사전조건:** 사업·공정관리에서 Task 계획됨
- **기본흐름:** ① 계획된 Task가 담당자별 작업카드로 자동 생성 → ② 담당자가 칸반(할일/진행중/검토요청/완료)에서 Drag&Drop으로 상태 변경 → ③ 카드 상세에서 진척률·코멘트·첨부·하위작업 등록 → ④ 검토자 검토 → ⑤ 완료 시 실적 반영.
- **대안/예외:** 2a. 지연 카드 자동 표시·조회·엑셀 다운로드.
- **사후조건:** 작업 실적 갱신(→UC-BRG-01). **관련:** REQ-KAN-01,02,03,04.

#### UC-BRG-01 WBS↔칸반 양방향 동기화 (계획↔실행 브리지) ★차별화
- **주액터:** SVC(동기화 엔진)/PM / **사전조건:** WBS Task와 작업카드가 매핑됨
- **기본흐름:** ① WBS Task 또는 칸반 카드 어느 쪽에서 진척 입력 발생 → ② 브리지 엔진이 단일 데이터로 정규화하여 양측 반영 → ③ 계획 관점은 S-Curve, 실행 관점은 번다운에 동시 반영 → ④ 단일 화면에서 계획↔실행 대조 제공.
- **대안/예외:** 2a. 충돌(동시 수정) → 최신 타임스탬프·감사로그 기록 후 재계산.
- **사후조건:** 계획·실행 지표 정합 유지(SSOT). **관련:** REQ-BRG-01,02 / 연계 REQ-SCH-05.

#### UC-QLT-02 산출물 다단계 결재
- **주액터:** PL→PM→CUS / **사전조건:** 산출물 제출됨, 결재선 정의
- **기본흐름:** ① 제출자가 산출물 제출 → ② 결재선(PL→QA→PMO→발주처 등)에 따라 순차 승인 → ③ 각 단계 승인/반려(반려 시 사유) → ④ 최종 승인 시 버전 확정·폴더 잠금.
- **대안/예외:** 3a. 반려 → 제출자에게 회신·재제출. 2a. 본인 대기건 일괄 승인.
- **사후조건:** 확정 산출물 무단변경 차단. **관련:** REQ-QLT-02,03,04.

#### UC-TST-02 / UC-TST-03 케이스 수행·결함 라이프사이클
- **주액터:** TST(수행)·TM(조치) / **사전조건:** 시험·케이스 구성됨(UC-TST-01)
- **기본흐름:** ① 시험자가 케이스 수행결과(성공/실패/보류/제외)·증적 등록 → ② 실패 시 멀티 결함 등록(클립보드 캡처 첨부) → ③ 담당자가 결함 조치(조치중→조치완료) 또는 비결함 검토요청 → ④ 시험자 동의/거부(거부 시 재오픈) → ⑤ 모든 결함 조치완료 시 재시험 → ⑥ 성공 시 미종결 결함 자동 CLOSED, 실패 시 선택적 재오픈/신규 결함.

- **사후조건:** 결함 라이프사이클·수행이력 영구 기록, RTM·KPI 갱신. **관련:** REQ-TST-02,03,04,05 / 연계 REQ-TST-07,08.

#### UC-COM-02 주간/월간 보고 5단계 워크플로
- **주액터:** PM(취합)/PL(작성) / **사전조건:** 보고 기간·대상 업무 설정
- **기본흐름:** ① PM이 보고 생성→작성요청 → ② 팀장이 본인 팀 섹션 작성(WBS·이슈·리스크·결함 진척 자동 수집) → ③ 작성중→검토(취합) → ④ 완료 → ⑤ DOCX 자동 출력.
- **대안/예외:** 1a. 작성요청·완료 시 담당자 자동 알림(→UC-NTF-01).
- **사후조건:** 표준 보고서 산출·배포. **관련:** REQ-COM-02,03.

#### UC-NTF-01 통합 알림 구독·발송

## 4. 핵심 업무 프로세스 흐름

### 4.1 진척 데이터 흐름 (계획 ↔ 실행, SSOT의 핵심)
```
[방법론/단계 정의] → [WBS Task 등록 + 가중치]
        │                       │ (자동 생성)
        ▼                       ▼
   [계획 기준선]          [칸반 작업카드]
        │                       │ 실적 입력(담당자)
        └──────► [브리지 엔진(정규화)] ◄──────┘
                        │ 단일 진척 데이터
        ┌───────────────┼────────────────┐
        ▼               ▼                ▼
  [S-Curve(계획)]   [번다운(실행)]   [EVM PV/EV/AC]
        └───────────────┴────────────────┘
                        ▼
              [대시보드 위젯 자동 집계]
```
어느 경로(WBS/칸반)로 입력하든 브리지 엔진이 단일 데이터로 정규화하므로, 계획 통제(S-Curve)와 실행 속도(번다운)가 같은 수치를 공유한다. 이 흐름이 uplusitPMS의 "계획은 Waterfall, 실행은 Agile, 데이터는 하나" 원칙을 구현한다.

### 4.2 요구사항 라이프사이클
```
[등록(미분류)] → [분류: 수용/대체/통합/제외] → [베이스라인 확정]
       └─ 확정 후 변경 → [사유·근거·승인자 필수] → [변경이력 영구보존] → [재확정]
                                                          │
                            [확정 요구만] ──► [프로그램 매핑] ──► [테스트 케이스 매핑] ──► [RTM 커버리지]
```

### 4.3 테스트 – 결함 라이프사이클
```
[케이스 수행] ─성공→ [PASS(검토확인)]
       │
       └─실패→ [멀티 결함 등록] → [조치중] → [조치완료] ┐
                     │                                   │
              [비결함 검토요청] ─시험자 동의→ [NOT_DEFECT]│
                     └─거부→ [재오픈]                    ▼
                                              [전 결함 조치완료] → [재시험]
                                                  ├─성공→ [미종결 결함 자동 CLOSED]
                                                  └─실패→ [선택적 재오픈 / 신규 결함]
```

### 4.4 산출물 결재 / 보고 워크플로
```
산출물: [제출] → [PL] → [QA] → [PMO] → [발주처] → [확정·폴더잠금]  (반려 시 사유→재제출)
보고서: [생성] → [작성요청] → [작성중(팀별)] → [검토(취합)] → [완료] → [DOCX]
```
두 워크플로 모두 단계 진입 시 통합 알림(UC-NTF-01)을 발생시킨다.

## 5. 개념 데이터 모델

### 5.1 주요 엔티티 (개념 수준)
> 물리 테이블·컬럼·타입은 설계 단계(solution-architect)에서 확정. 여기서는 엔티티·핵심 속성·관계만 정의한다.

| User | 사번, 성명, 계정, 상태 | 전역 |
| Organization(Team) | 팀명, 상위조직 | 계층 |
| Role / PermissionGroup | 역할코드, 권한집합 | RBAC, 역할코드→권한 매핑 |
| ProjectMember | Project×User, 역할, 투입기간, 공수 | 투입인력 |
| Code / Holiday / MetaStandard | 공통코드, 공휴일, 표준단어·도메인·용어 | 기준정보 |
| WBSTask | 단계, 업무, 계획기간, 실적기간, 가중치, 진척률, depth, 선후행 | 일정 핵심 |
| WorkCard | 상태(할일/진행중/검토/완료), 진척률, 담당자, 검토자 | 칸반, WBSTask와 매핑(브리지) |
| Requirement | 유형(기능/비기능), 상태, 베이스라인버전 | 요구 |
| RequirementChange | 사유, 근거문서, 승인자, 일시 | 변경이력 |
| Program | 분류(화면/배치/IF/리포트/공통/이행), 계획·실적 | 개발진척 |
| RequirementMap | Requirement×Program, Requirement×TestCase | 추적 매핑 |
| Deliverable / DeliverableVersion | 산출물, 버전, 잠금여부 | 품질 |
| Approval / ApprovalLine | 결재선, 단계, 상태, 사유 | 다단계 결재 |
| TestRun / Scenario / TestCase | 시험종류(UT/IT/UAT/AT), 차수, 계층, 경로 | 테스트 |
| Execution | 결과(성공/실패/보류/제외), 증적, 기간 | 수행이력 |
| Defect / DefectHistory | 심각도, 유형, 상태, 첨부, 상태변경이력 | 결함 |
| Issue / Risk | 상태, 우선순위, 영향도, 대응방식 | 위험 |
| MeetingRoom / Reservation / SecurityPledge | 회의실, 예약, 보안서약 | 자원 |
| Meeting / ActionItem | 회의록, 담당자, 이행상태 | 의사소통 |
| Report / ReportSection | 주간/월간, 워크플로 상태, 팀별 섹션 | 보고 |

| Board / Post | 게시판, 글, 고정·필독 | 협업 |
| Message / Channel | 쪽지·채널, 수신자, 읽음 | 메신저 |
| Calendar / Event / Milestone | 일정, 이벤트유형, 마일스톤 | 캘린더 |

| AuditLog | 대상, 행위(CUD), 행위자, 일시 | 감사 |

### 5.2 핵심 관계 (텍스트 ERD)
```
Project 1──N ProjectMember N──1 User
Project 1──N WBSTask 1──N WBSTask(선후행 자기참조)
WBSTask 1──1..N WorkCard            ← 계획↔실행 브리지
Project 1──N Requirement 1──N RequirementChange
Requirement N──M Program  (RequirementMap)
Requirement N──M TestCase (RequirementMap)         → RTM
TestRun 1──N Scenario 1──N TestCase 1──N Execution
TestCase 1──N Defect 1──N DefectHistory
Project 1──N Deliverable 1──N DeliverableVersion
Deliverable 1──N Approval (ApprovalLine 단계)
Project 1──N {Issue, Risk, Meeting(1─N ActionItem), Report(1─N Section)}
User 1──N Notification ; Event ─발생→ Notification (Subscription 매핑)
모든 CUD ─기록→ AuditLog
```
RequirementMap이 요구↔개발↔테스트를 잇는 추적성의 중심 엔티티이며, WBSTask↔WorkCard 매핑이 계획↔실행 단일화의 중심이다.

### 6.2 핵심 화면 전환 흐름
- **대시보드 → 드릴다운:** 위젯 클릭 → 해당 모듈 상세(진척/이슈/결함/결재) 직접 진입.
